# XOR Many-Time Pad Cryptanalysis Suite

Цей проєкт представляє собою інструмент для статистичної реконструкції ключового потоку (keystream) потокових шифрів. Він демонструє фундаментальну вразливість повторного використання ключа в криптографії (Many-Time Pad attack).

## 1. Фізика процесу: Як працює шифр XOR
Шифрування виконується за допомогою побітової операції "Виключне АБО" (XOR) між текстом (Plaintext, $P$) та ключем (Key, $K$), утворюючи шифрограму (Ciphertext, $C$):

$$C = P \oplus K$$

Для дешифрування операція повторюється, оскільки XOR є симетричною функцією:

$$P = C \oplus K$$

Якщо ключ генерується випадковим чином і дорівнює довжині повідомлення (ідеальний One-Time Pad), система є математично незламною, оскільки шифротекст має максимальну ентропію.

## 2. Вразливість повторного використання ключа
Критична помилка виникає, коли один і той самий ключ використовується для шифрування декількох різних повідомлень. Якщо ми маємо дві шифрограми ($C_1$ та $C_2$), зашифровані однаковим ключем ($K$), перехоплювач може виконати операцію XOR між ними:

$$C_1 \oplus C_2 = (P_1 \oplus K) \oplus (P_2 \oplus K) = P_1 \oplus P_2$$

В результаті ключ взаємознищується. Ентропія різко падає, і ми отримуємо прямий логічний зв'язок між двома відкритими текстами. Оскільки природна мова (англійська) має жорсткі статистичні та граматичні правила, ми можемо відновити текст, просто перебираючи ймовірні значення ключа та оцінюючи "адекватність" результату. Чим більше повідомлень (глибина матриці), тим легше виокремити корисний сигнал із шуму.

## 3. Алгоритм злому та Обмеження (Constraints)
Скрипт використовує метод максимізації цільової функції (Fitness Function) для побайтового підбору ключа. Алгоритм не просто перебирає символи, а пропускає кожен варіант через каскад інженерних фільтрів:

1. **Range Filter (Жорстке відсікання):** Байт ключа визнається недійсним, якщо він генерує недрукований ASCII-символ.
2. **Frequency Filter (Статистика):** Нарахування балів на основі апріорної ймовірності символів англійської мови (найвища вага у пробілу та літери 'e').
3. **Continuity Filter (Біграми):** Бонуси за природні поєднання літер (наприклад, 'th', 'he', 'qu'), що забезпечує зв'язність слів.
4. **Case Integrity (Контроль регістра):** Жорсткий штраф за переходи регістра "мала -> ВЕЛИКА" та "ВЕЛИКА -> ВЕЛИКА" всередині слів. Це ліквідує колізію 0x20 — типову помилку при XOR, коли алгоритм плутає регістр літер.
5. **Noise Reduction (Пригнічення шуму):** Штрафи за неалфавітні символи всередині тексту. Алгоритм свідомо "відкидає" цифри та специфічну пунктуацію (#, ^, &), надаючи пріоритет літерам, щоб очистити сигнал.

## 4. Обмеження системи
Незважаючи на багатофакторний аналіз, цей інструмент є статистичним, а не детермінованим. Він може допускати локальні помилки через наступні причини:

* **Закон великих чисел:** Алгоритм потребує глибокої вибірки. Якщо для однієї позиції ключа ми маємо лише 3-5 повідомлень (або короткі тексти), статистика не встигає спрацювати, і випадковий збіг (шум) може набрати більше балів, ніж правильна літера. **Чим більше повідомлень перехоплено, тим вища математична точність відновлення.**
* **Аномальна лексика:** Якщо у вихідному тексті багато абревіатур (наприклад, `NASA`), нетипових імен чи спеціальних символів, словникові фільтри та контроль регістра дадуть збій, оскільки текст не підпорядковується стандартним правилам мови.
* **Сліпі зони:** Якщо в певній колонці (на певній позиції $i$) у всіх повідомленнях стоять пробіли або однакові символи, система не матиме достатньо контексту для правильного визначення ключа.
